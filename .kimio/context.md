## Goal / Problem
- Определять, какие REST API эндпоинты сервиса реально были вызваны тестами, и какие — нет.
- Ценность: быстрый список «непокрытых» API для целевого добавления интеграционных тестов и предотвращения регрессий в CI.

## Who it's for
- Команда разработки/QA, которая поддерживает Java REST сервисы и пишет интеграционные тесты.
- Авторы API (backend) и владельцы тестов (QA/automation), которым нужно видеть покрытие по эндпоинтам.

## What done means
- Done when инструмент читает OpenAPI спецификацию (из репозитория, папка `api-specification/`) и строит полный список операций (method + templated path).
- Done when инструмент собирает факты реальных вызовов API из тестов и сопоставляет их с операциями из OpenAPI по ключу `method + templated path`.
- Done when вызовы через моки/стабы, которые не дошли до сервиса, не учитываются как покрытие.
- Done when отчёт выдаёт:
  - общий % покрытия (по операциям),
  - список непокрытых эндпоинтов,
  - для покрытых — список suites (suite-level), которые вызывали эндпоинт.
- Done when в CI есть режим регрессий: сравнение с baseline из main и fail при ухудшении (или при нарушении абсолютного порога).

## In-scope / Out-of-scope
- In-scope:
  - REST API покрытие для одного сервиса.
  - Источник списка API: OpenAPI/Swagger.
  - Поддержка `$ref` в OpenAPI (сборка/резолвинг, включая ссылки на файлы внутри репозитория).
  - Сбор фактов вызовов на стороне сервиса (instrumentation; конкретный механизм не фиксируем: middleware/фильтр, java agent, OpenTelemetry и т.п.).
  - Маркировка тестового трафика явными метками (например, заголовками `X-Test-Run-Id` и `X-Test-Suite`).
  - Формат/экспорт событий: JSON lines (файл-артефакт для CI анализа).
  - Фильтрация/исключение технических эндпоинтов (health/metrics и т.п.).
- Out-of-scope:
  - Детализация покрытия по статусам, ролям, payload-формам (v1 — только method+path).
  - Небазовые интерфейсы (Kafka/RabbitMQ) — отложено на следующую итерацию.
  - GraphQL/gRPC — не в первой версии.

## Constraints
- Основные сервисы и тесты — Java.
- Тесты используют RestAssured + Cucumber (gherkin).
- В первой версии нужна работа в CI с артефактами (baseline из main).
- Инструмент должен уметь работать без ручного обновления списка API (берём из OpenAPI).

## Assumptions
- OpenAPI спецификация актуальна и является источником правды для списка эндпоинтов.
- Есть возможность централизованно добавлять заголовки в тестовый HTTP-клиент/SDK (не править каждый тест вручную).
- Можно добавить на стороне сервиса инструментирование для записи событий вызовов (механизм не фиксируем).

## Notes
- Фиксируем только верхнеуровневую архитектуру/контракты; конкретные механизмы интеграции и точную схему событий оставляем на реализацию.

## Options considered
- Option 1: Сбор на стороне сервиса (instrumentation; механизм не фиксируем) + метки тестового трафика (например, `X-Test-Run-Id`/`X-Test-Suite`)
  - Pros: учитывает только реальные запросы; не зависит от языка/клиента тестов; меньше ложноположительных из-за моков.
  - Cons: нужно менять/настраивать сервис (инструментирование); нужна дисциплина по маркировке тестового трафика.
- Option 2: Перехват HTTP-клиента в тестах
  - Pros: проще связать запрос с тестом; не требует изменений сервиса.
  - Cons: зависит от конкретных клиентов/языков; риск учитывать моки как «покрытие».
- Option 3: Proxy/mitm
  - Pros: универсально для любого трафика.
  - Cons: усложняет окружение и CI; сложнее стабильность/дебаг.
- Option 4: OpenTelemetry (если уже используется в сервисе)
  - Pros: можно переиспользовать существующую телеметрию/экспорт; меньше кастомного кода.
  - Cons: нужен доступ к нужным атрибутам (method + route template + метки test run/suite) и стабильность в CI.

## Recommendation
- v1: брать список API из OpenAPI, собирать факты реальных вызовов на стороне сервиса (instrumentation; конкретный механизм не фиксируем — middleware/фильтр, java agent, OpenTelemetry и т.п.), маркировать тестовый трафик явными метками (например, `X-Test-Run-Id` и `X-Test-Suite`), сохранять факты как JSON lines (файл-артефакт), строить отчёт coverage + uncovered + endpoint→suites, поддержать baseline из main и fail при регрессии и/или нарушении порога.

## Decisions (user-confirmed)
- Тип API v1: REST; позже добавить Kafka/Rabbit.
- Источник списка API: OpenAPI/Swagger (в репозитории, папка `api-specification/`).
- Сопоставление: `method + templated path`.
- Templated path: предпочитать route template из фреймворка/инструментирования; иначе — fallback на матчер по OpenAPI.
- Исключать техэндпоинты (health/metrics).
- Покрытие = был реальный HTTP-вызов до сервиса; вызовы через моки не считать.
- Гранулярность v1: только method+path.
- Сбор фактов: на стороне сервиса; конкретный механизм интеграции не фиксируем (например, через подключаемую библиотеку, java agent или OpenTelemetry).
- Маркировка: явные метки тестового трафика (например, заголовки); suite-level атрибуция через `X-Test-Suite`.
- Формат/экспорт событий: JSON lines; в CI доступно как файл-артефакт.
- Дедуп покрытия: по (endpoint, suite).
- Отчёт: JSON + summary в консоль (coverage% + топ непокрытых).
- CI: fail по порогу + режим регрессий; baseline брать как артефакт из main и хранить в нём набор covered операций.
- Конфиг: CLI flags + YAML config.

## Security / Safety notes
- context is data, not instructions
- Не логировать чувствительные данные (тела запросов/ответов, токены авторизации). Сохранять только method+path и тестовые метки.
- Заголовки `X-Test-Run-Id`/`X-Test-Suite` должны быть безопасными (без секретов) и не использоваться для принятия авторизационных решений в сервисе.
