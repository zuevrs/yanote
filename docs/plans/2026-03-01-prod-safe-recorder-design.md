## Prod-safe yanote recorder (design)

**Цель:** сделать `yanote-recorder-spring-mvc` безопасным при подключении в production: ничего не пишет и не влияет на запросы, пока его явно не включили.

### Контекст

`yanote-recorder-spring-mvc` пишет события HTTP вызовов в `events.jsonl`, чтобы потом `yanote-cli` посчитал покрытие API по OpenAPI.

Текущий риск: стартер включён по умолчанию и, если его поставить в prod “как есть”, он начнёт писать прод-трафик в файл, а при проблемах записи может повлиять на стабильность.

### Принятое решение (recommended)

- **Disabled-by-default**:
  - `yanote.recorder.enabled` по умолчанию `false`
  - авто-конфигурация применяется только если `yanote.recorder.enabled=true` задан явно
- **Fail-safe запись**:
  - ошибки записи событий не должны падать наружу и ломать обработку запроса
  - при ошибке — залогировать предупреждение и “дропнуть” событие

### Не делаем сейчас (YAGNI)

- сэмплирование и canary-режимы
- отдельные sinks (OTel/Kafka/HTTP collector)
- буферизация/батчинг/асинхронная запись
- режим “писать только по специальным заголовкам” (header-gating) — оставляем как возможное развитие

### Success criteria

- Подключение стартера в prod **ничего не меняет**, пока `yanote.recorder.enabled=true` не задан явно.
- При включении и невозможности писать файл (нет прав/диск/FS) запросы продолжают работать (нет 5xx из-за рекордера).
